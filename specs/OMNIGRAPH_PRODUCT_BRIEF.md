# OmniGraph: Product Brief (Revised v2)

**Date:** 2026-04-23  
**Status:** Ready to build

---

## 定位

**Vibe Coding时代的虚拟CTO技能（安全版）**

> **“你负责想，我帮你别选错，Claude Code负责做。”**

以 Hermes / OpenClaw skill 形态交付：

- 用户输入业务需求  
- 系统通过轻量引导抽取关键约束（≤3问）  
- 输出一个**安全可落地的技术方案 + 明确禁止项 + 成本提示 + TDD**  
- 可直接驱动 Write Code Skill 实现  

---

## 核心价值

### 用户真实问题

Vibe coder最大瓶颈不是“不会写代码”，而是：

- 不知道什么时候 over-engineering  
- 不知道哪些技术现在不该用  
- 不知道方案是否适合 solo dev  
- 不知道成本是否会失控  

当前LLM（如Claude Code）：

- 倾向给出“能做”的方案  
- 但不会主动避免复杂或错误选型  

---

### OmniGraph 提供

> ✅ **不会后悔的默认方案（Safe Default）**

而不是：

- ❌ 最先进方案  
- ❌ 最复杂架构  

---

### 核心原则

- 默认保守（prefer simple over powerful）  
- 明确否定（明确告诉用户不要用什么）  
- 渐进升级（progressive enhancement）  

---

## 核心体验（CX）

### 整体流程

```

用户需求
↓
Step 1：系统默认猜测（给出初始方案）
↓
Step 2：2–3个关键问题（抽取约束）
↓
Step 3：输出（推荐 + 禁止 + 成本 + TDD）

```

---

### Step 1：默认猜测（降低认知负担）

```

你可能想做：
→ 一个简单AI客服（MVP）

可以直接用：
→ OpenAI API + SQLite

如果不对，我问你2个问题帮你修正👇

```

---

### Step 2：关键问题（最多3个）

- 是否需要长期记忆（memory）  
- 是否有多步骤复杂流程（agent）  
- 是否 solo dev（复杂度上限）  

---

### Step 3：输出结构（核心价值）

```

推荐方案（安全版）
复杂度：Low
适合：solo dev

不要用（现在）：

* LangGraph（过度复杂）
* 向量DB（当前不需要）

成本：

* API：低（<$20/月）

升级路径：

* 用户量上来后 → Postgres / memory

TDD：
...

```

## 技术架构

```

用户业务需求（自然语言）
↓
/architect skill（OmniGraph核心）
↓
┌────────────────────────────┐
│  规则引擎（核心决策层）     │
│  - 约束判断                │
│  - 推荐方案               │
│  - 禁止项输出             │
└────────────────────────────┘
↓
┌────────────────────────────┐
│  Graphify MCP              │  ← 工具结构理解
│  LightRAG                  │  ← 社区经验/案例
│  entity_registry.json      │  ← GitHub URL锚点
└────────────────────────────┘
↓
结构化输出：

* 推荐方案（安全默认）
* 禁止项（核心）
* 理由（知识增强）
* TDD模板
  ↓
  Write Code Skill → 实现

```

---

## 关键设计

### 1. 决策与知识分层

- **规则引擎 = 决策核心（稳定、可控）**
- **Graphify / LightRAG = 解释增强（辅助）**

---

### 2. 两图谱互通设计（保留）

#### 身份锚点（静态层）
- 使用 GitHub URL 作为全局唯一ID  
- `entity_registry.json` 存储映射关系  

#### 查询互通（运行时）
- Graphify 与 LightRAG 并发查询  
- 生成层进行语义合并  

#### 质量保障
- 优先校验 god nodes（高连接度工具）  
- 长尾节点允许误差  

---

### 3. 规则驱动（核心差异）

不是依赖RAG自动推理，而是：

> **用规则做决策，用数据做解释**

## 构建路线图

### Phase 1 · MVP（2–3天）

- 定义20条反坑规则  
- 实现最小状态机（INIT → GUESS → ASK → OUTPUT）  
- 实现 /architect skill  
- 固定输出模板  
- README（强调避免踩坑）  

---

### Phase 2 · 内容增强

- Graphify接入  
- entity_registry.json建立  
- LightRAG基础接入  

---

### Phase 3 · 社区增强

- 微信 / 知乎内容采集  
- 案例抽取（踩坑总结）  
- 输出解释增强  

---

### Phase 4 · 优化

- 提问优化（减少到2个）  
- 默认猜测优化  
- 场景模板（客服 / 自动化 / 内容生成）  

---

## MVP完成标准

```

✓ 用户输入：“做一个AI客服”
✓ 系统直接给出一个可用简单方案
✓ 最多提问2个问题
✓ 输出包含：

* 安全方案
* 明确“不要用什么”
* 成本提示
* 升级路径
  ✓ 5分钟内开始写代码

```

---

## 技术风险

1. 规则质量不足（核心风险）  
→ 手工打磨规则  

2. 输出过于保守  
→ 提供升级路径  

3. KOL内容噪音  
→ 仅用于解释
