# Quick 260524-tk5b — Databricks SDK silent-hang 终结战（第三个文件）

## 背景

前置 quick `260524-tk5` 装了仪表（T1）+ 拉了 inner timeout（T3 — 在 `kg_synthesize.py` 把 `rag.aquery()` 包了 `asyncio.wait_for(timeout=150)`），但 6/6 验证全失败：

| # | 标准 | 结果 |
| - | --- | --- |
| 1 | `status == "done"` | PASS |
| 2 | `fallback_used == false` | **FAIL** (true — 落到 FTS5) |
| 3 | `markdown.Length > 1000` | **FAIL** (86 — 只剩兜底文案) |
| 4 | markdown 含 `![alt](url)` | **FAIL** (0 张图) |
| 5 | wall-clock < 180s | **FAIL** (249s) |
| 6 | `c1_after_aquery` AND `kg_after_aquery` 都打 | **FAIL** (两条都没打) |

T3 仪表证据（`.scratch/uvicorn-T3.log`）显示挂死是**确定性的**：attempt 1 在 LightRAG dispatch LLM 调用后挂 150s → 内层 TimeoutError 触发 → sleep 5s → attempt 2 **在同一个边界以同样的形态再挂死一次** → outer 240s wrapper 抢先 cancel attempt 2。不是 worker queue 阻塞这种"过一会儿就好"的瞬态问题，是同一个 SDK HTTP 调用同样地挂两次。

完整失败记录看 `.scratch/synthesis-c1-fix-2026-05-24-2131.md` 的 "Diagnosis: this is the plan's risk note boundary case" 段（plan Risks line 253 的 b1+b2 straddle 风险被命中）。

## 真根因

`databricks-deploy/lightrag_databricks_provider.py` 行 94-100 的 LLM 调用：

```python
loop = asyncio.get_running_loop()
resp = await loop.run_in_executor(
    None,
    lambda: w.serving_endpoints.query(
        name=KB_LLM_MODEL, messages=messages
    ),
)
return resp.choices[0].message.content
```

三个独立缺陷叠加：

1. **`serving_endpoints.query()` 没传 `timeout` kwarg** — Databricks SDK 文档说 default client-side HTTP timeout 是 120s，但实测 230s 才被外层 wrapper cancel 掉，强烈暗示 default 在 streaming/SSE 路径上**没有真正生效**。底层 `requests.post(stream=True).iter_lines()` 卡在 `socket.recv()`，TLS read 没收到 FIN（很可能 EDC corp 的 Cisco Umbrella TLS-intercept 偶尔吞掉最后那个 SSE chunk，让 generator 永不关闭）。
2. **`WorkspaceClient` 实例化没传 `Config(http_timeout_seconds=...)`** — `databricks-deploy/_db_client.py` 的 `get_databricks_client()` 把 kwargs 透传到 `WorkspaceClient`，但调用方 `lightrag_databricks_provider.py:71` 直接 `w = get_databricks_client()` 没塞 config。即使 default 120s 真的生效，也没办法显式提高/降低。
3. **`messages=...` 没传 `max_tokens`** — Databricks Foundation Models 走的是 Anthropic Claude API 的 OpenAI-compatible 兼容层，**Claude Sonnet 4 默认 `max_tokens=1000`**。13198-char 中文 long_form prompt + KG context 大概率会把 prompt+thinking 加起来超过 64K 输入预算，模型会用更长的输出来覆盖整个上下文 — 加上没有 `max_tokens` 上限，模型就一直 stream 直到 server-side 自然 stall 或者上游网络 keep-alive 断掉。

`asyncio.wait_for(timeout=150)` 在 `kg_synthesize.py` 里挡不住底层 daemon thread —— `loop.run_in_executor(None, ...)` 把 SDK 调用扔到 default thread pool，一旦 daemon thread 卡在 `socket.recv()` 上，asyncio task 取消的语义只能让 awaiter 重新得到控制，**socket 还是会继续等下去**。这是为什么 attempt 2 同样地挂 — sleep 5s 期间 attempt 1 的 daemon thread 还在挂着，attempt 2 起一个新 daemon thread 走同一条 SSE 路径，落到同样的 stall。

## 修复 — Option C（user 选定：config-level + call-level + max_tokens）

三处协同：

### 修 1：`databricks-deploy/_db_client.py` — default config

`get_databricks_client()` 返回的 `WorkspaceClient` 默认带 `Config(http_timeout_seconds=120)`。调用方不传 `config=` 时也吃这个默认；显式传 `config=` 的覆盖默认。

### 修 2：`databricks-deploy/lightrag_databricks_provider.py` LLM 路径

`serving_endpoints.query()` 显式加 `timeout=120.0`（call-level）+ `max_tokens=63999`（用户指定 = 模型 max 64000 - 1）。前者短路 daemon thread（HTTP-level abort），后者强制服务端在生成 ≤63999 token 时主动关闭 SSE stream。

### 修 3：`databricks-deploy/lightrag_databricks_provider.py` embedding 路径

行 118-127 的 embedding 调用同样 missing-timeout（同一个 bug 形态）。加 `timeout=60.0`。embedding 不需要 `max_tokens`（不是生成调用）；60s 足够覆盖 1024-dim Qwen3-embedding 的最坏延迟。

### 修 4：保留 tk5 的仪表 + inner timeout

`kb/services/synthesize.py` 和 `kg_synthesize.py` 的 T1+T3 改动从 tk5 滚进来一起 commit（防御纵深 — call-level 120s + config 120s + asyncio.wait_for 150s 三层时序对齐：SDK 在 120s 把 HTTP socket 干掉 → 异常冒泡到 `loop.run_in_executor` await → asyncio task 在 150s budget 用尽前就拿到正常异常 → retry loop 进 attempt 2）。

## Risks

- **`Config` + `profile` kwarg 互斥风险.** `WorkspaceClient(profile=..., auth_type='pat', config=...)` 三个 kwarg 同时传可能在某些 SDK 版本撞 `ValueError: cannot set both ... and config`。Mitigation: T1 done criteria 的 import sanity 命令直接走 `get_databricks_client()`（含 profile 路径）— 如果撞 ValueError，立刻知道；fallback 是去掉 default config，只在 caller 显式传 — 但这就回到 Option B 而非 C，需要重新和用户确认。

- **Databricks SDK `max_tokens=63999` 上限.** Anthropic Claude Sonnet 4.6 sync Messages API 最大 64,000 output tokens；Databricks Foundation Model serving 可能在更低层级拒绝（例如 endpoint config 限制）。Mitigation: T2 import sanity 之后做一个最小生成调用（5-token prompt + max_tokens=63999），看是否抛 4xx。如果抛 → 回退到 max_tokens=8000（多数生产 long-form 答案的实际所需）+ 在 memo 里记录这个 ceiling。

- **Embedding `timeout=60.0` 不够.** 1024-dim Qwen3 单 batch 60s 通常够，但 LightRAG 的 hybrid 检索一次会触发多次 embedding 调用（每个 query 的实体词嵌入）。Mitigation: T3 验证 #5（wall < 180s）会兜底 — 如果 embedding 总时间挤掉了 LLM 预算，会在那一项失败，自动 surface。

- **`kb/services/synthesize.py` + `kg_synthesize.py` 的 tk5 改动没 commit.** 它们是 dirty working tree，`git diff` 显示能看到，但 git history 里完全没有。一旦这次 quick 也失败、user 让 revert 全部，需要分两次 revert。Mitigation: T4 把这两个文件一并 stage 进同一个 commit（已在 staged-set audit 中列出），让"通过"和"修复"原子化。

- **commit drift.** `.planning/STATE-Aliyun-Ingest-Migration-v1.md` / `requirements.txt` / `.planning/phases/aim-4-daily-sync/*` 是 aim-4 phase 的工作，**严禁** stage 进这次 commit。Mitigation: T4 step 2 显式用 `git diff --cached --stat` 审计；只有恰好 6 个文件才允许进 commit。

- **MCP 工具调用入坑（硬约束 #7）.** 这次 quick 不需要 MCP，但如果 T3 验证里要查 Databricks endpoint 状态，可能想用 `mcp__databricks-mcp-server__execute_sql` — **必须主会话调用**，不在 sub-agent 里 dispatch。
