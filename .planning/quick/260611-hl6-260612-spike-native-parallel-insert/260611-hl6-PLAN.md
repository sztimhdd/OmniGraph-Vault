# 260611-hl6 PLAN — 260612-spike-native-parallel-insert

**Quick ID:** 260611-hl6
**Goal:** Measure whether LightRAG's NATIVE `ainsert(list)` concurrency path (one call with `max_parallel_insert` semaphore) yields meaningful speedup vs serial single-doc `ainsert()` calls on Aliyun prod-parity environment.

**Status:** EXECUTING

---

## Why this quick exists

Prior probes (mz1/pwl/u17/v3) all tested hand-written `asyncio.gather(ainsert(d1), ainsert(d2))` — stacking a SECOND concurrency layer on top of LightRAG's own `pipeline_status` singleton. This caused "Another process is already processing the document queue" + "Duplicate document detected" → dedup gate short-circuit → 0.005s ghost result.

Source-read on Aliyun LightRAG 1.4.16 revealed that `ainsert(input: list[str])` already uses an internal `asyncio.Semaphore(self.max_parallel_insert)` around per-doc `process_document` coroutines (lightrag.py:1871). Merge stage has deadlock-prevention via `sorted_key_parts` (operate.py:750) + `sorted_edge_key = tuple(sorted(...))` (operate.py:2562). This is the path we should have been testing all along.
