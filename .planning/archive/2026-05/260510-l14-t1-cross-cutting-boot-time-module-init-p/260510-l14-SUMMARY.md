## Risk events

### Deviation 1 — `tests/unit/test_lightrag_llm.py` required surgical patch (W1)

**Plan claim:** "Existing tests in `tests/unit/test_lightrag_llm.py` (4 tests using `from lightrag_llm import deepseek_model_complete` + 1 testing `from lib.llm_deepseek import`) MUST continue to pass — those use full-path imports."

**Reality:** 8 of 17 tests in `test_lightrag_llm.py` patch `lib.llm_deepseek._client` directly (e.g., `with patch.object(ld._client.chat.completions, "create", mock_create)`). After W1's lazy fix, `_client` starts as `None` and is constructed only on first `_get_client()` call. The eight patch-based tests fail with `AttributeError: 'NoneType' object has no attribute 'chat'` until the lazy client is materialized.

**Fix applied:** Surgical patch to `tests/unit/test_lightrag_llm.py`:
1. Autouse fixture `_ensure_deepseek_key` now pre-builds the lazy client via `_ld._get_client()` so subsequent `patch.object(ld._client...)` invocations work.
2. `test_deepseek_model_env_override` re-imports the module and so must rebuild the lazy client itself (added `ld._client = None; ld._get_client()`).
3. `test_missing_api_key_raises_runtime_error` rewritten to assert the deferred raise on first `_get_client()` call (was: assert raise at module import).

**Justification:** The plan's "MUST continue to pass" was incompatible with the strict letter of the lazy-init implementation. The patch is minimal (3 hunks, +20/-7 LOC) and preserves test intent: each test still verifies the same dispatch behavior as before. The new `test_missing_api_key_raises_runtime_error` accurately reflects the new contract (deferred raise) instead of pinning the old eager-raise.

### Deviation 2 — Line-ending normalization required after Edit tool churn (W2)

**Symptom:** After editing 6 CLI scripts via the `Edit` tool, `git diff --stat` showed 679 / 477 / 113 / etc. line-count changes — the entire files appearing as "all lines changed".

**Cause:** `Edit` writes LF line endings; the original Hermes-deployed files used a mix of LF (e.g., `enrichment/fetch_zhihu.py`, `ingest_github.py`, `omnigraph_search/query.py`) and CRLF (`enrichment/merge_and_ingest.py`, `query_lightrag.py`, `run_uat_ingest.py`).

**Fix applied:** A small Python helper normalized each file back to its HEAD line-ending form (`git show HEAD:<file>` → checked for `\r\n` presence → matched on the working copy). After normalization, `git diff --stat` showed clean surgical diffs.

**Justification:** Surgical-changes principle (CLAUDE.md "Highest Priority Principle 3"): preserve existing style. The mixed line-endings are an artifact of legacy Windows/Linux contributors and not something this quick should normalize across the repo (that would explode the diff and obscure the real changes). Local fix per file kept the commit reviewable.
