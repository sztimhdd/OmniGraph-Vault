**Schema-shift caveat (executor MUST honor):**
The existing `UNIQUE(article_id, topic)` constraint allows multiple rows per article (one per topic).
The new `UNIQUE(article_id)` constraint **collapses** that to one row per article. This is INTENTIONAL
per user spec — the production code already treats classifications as one-row-per-article (the
`MAX(classified_at)` subquery proves the cron query already coalesces duplicates). The dedup step
in migration 004 keeps `MAX(rowid)` per `article_id`, matching cron query semantics. Executor must
NOT add a UNIQUE(article_id, topic) preserving compromise — that defeats the spec.
