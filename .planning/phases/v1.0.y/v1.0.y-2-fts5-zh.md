# v1.0.y-2 — Diagnose + fix zh-CN 27-char FTS5 fallback zero-hit

## Goal

Restore non-zero hit rate on the `_fts5_fallback` path for zh-CN
short-medium queries (representative case: the 27-char query that
surfaced in `.scratch/260525-kdb-smoke-longform.log` returning
`confidence='no_results'` even though `articles_fts` contains
substantively related zh-CN articles).

This phase is a **single phase with two sequential tasks**:

- **Task 1 — Diagnose** the zero-hit root cause (one of: tokenizer,
  index drift, query-side normalization, or FTS5 SQL shape).
- **Task 2 — Fix** based on Task 1's findings.

Per Q2 (locked): NOT split into two phases. Task 2 is gated by Task 1
producing a clear root-cause classification with a corresponding fix
sketch — if Task 1 is inconclusive, halt and escalate (do NOT
speculate-fix multiple hypotheses in the same commit).

This phase has **no upstream blocker** (parallelizable with v1.0.y-1).

## Background

### Symptom (verbatim from `.scratch/260525-kdb-smoke-longform.log` cross-reference)

The 2026-05-25 long_form smoke produced a **double miss**: outer 60s
preempted C1 AND the FTS5 fallback found zero hits for the
(zh, 27-char) question. The empty markdown is FTS5's "I tried, nothing
matched" — separate from the timeout root cause (which v1.0.y-0/-1
already address). Per REPORT.md §"_fts5_fallback returns confidence='no_results' not 'fts5_fallback' — why?":

> The FTS5 fallback exists to provide *some* answer when C1 fails.
> `confidence='no_results'` (vs `'fts5_fallback'`) means **FTS5 also
> found zero matching articles** for the query terms (per
> `kb/services/synthesize.py:_fts5_fallback` semantics — when the
> FTS5 SELECT returns zero rows, the helper sets
> `confidence='no_results'` and writes empty `markdown`).

### Why this is non-trivial (4 candidate root causes)

| # | Hypothesis | How to test | Fix shape |
|---|---|---|---|
| 1 | **Tokenizer** — FTS5 default `simple` / `unicode61` does NOT split CJK runs into terms; entire 27-char phrase is one "token" that does not match anything in the column tokens. | Inspect `articles_fts` schema for `tokenize=` clause; run `SELECT * FROM articles_fts WHERE articles_fts MATCH '<single-char>'` to see if anything matches. | Switch tokenizer to `unicode61 remove_diacritics 1 separators "..."` with CJK-aware separators, OR adopt `jieba_tokenizer` (requires extension), OR pre-tokenize at write time with explicit space-separated zh tokens column. |
| 2 | **Index drift** — `articles_fts` was rebuilt at some point against a different base; row count or content does not match `articles` table. | `SELECT COUNT(*) FROM articles_fts` vs `SELECT COUNT(*) FROM articles` for canonical Aliyun DB. Check timestamps. | `INSERT INTO articles_fts(articles_fts) VALUES('rebuild')` after Task 1 confirms drift. |
| 3 | **Query-side normalization** — `_fts5_fallback` builds the MATCH expression naively (e.g., uses raw query text or escapes incorrectly), so even a tokenized index misses. | Read `_fts5_fallback` source; run the exact MATCH expression against sqlite3 CLI manually. | Patch query construction (escape, OR-split tokens, prefix wildcards). |
| 4 | **FTS5 SQL shape** — column list / rank / LIMIT cuts off legit hits prematurely; or the helper SELECTs columns that don't exist. | Add temp logging in `_fts5_fallback` (or run a parallel manual probe SQL) to compare hit rate query-vs-MATCH. | Adjust SQL projection / LIMIT / ORDER BY rank. |

These are not mutually exclusive — likely 1 + 3 combined (CJK
tokenizer absent AND query construction naive). Task 1 must classify
which combination.

halt + escalate to user before proceeding to Task 2.
