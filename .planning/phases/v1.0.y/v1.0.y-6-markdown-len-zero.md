# v1.0.y-6 — `markdown_len=0` on long_form despite `confidence=kg`

## Goal

Diagnose and fix the residual `markdown_len=0` failure surfaced by the
v1.0.y-0 C1 timeout fix. After the timeout drift was closed, long_form
`POST /api/synthesize` on Databricks now runs to completion (98s wallclock,
`confidence=kg`, `fallback_used=False`, `error_field=None`) but the final
`result.markdown` field is empty.

## Background — surfaced by v1.0.y-0 close

v1.0.y-0 fixed the outer `asyncio.wait_for(60s)` preempt that was killing
long_form before LightRAG hybrid could complete. Pre-fix, the run failed
at wallclock=64s with `confidence=no_results + error='C1 timeout'`. Post-fix
(`.scratch/260525-kdb-smoke-longform.log` lines 71-133):

```
wallclock_s=98.26          <-- past the old 60s preempt
confidence=kg              <-- LightRAG hybrid returned
fallback_used=False        <-- no FTS5 fallback
error_field=None           <-- no "C1 timeout"
markdown_len=0             <-- new failure surface
markdown_first200=         <-- empty
```

This failure mode is **causally distinct** from the C1 timeout:
- Pre-fix path: wait_for fires → `_fts5_fallback` → `confidence=no_results +
  error="C1 timeout" + markdown_len=0`
- Post-fix path: wait_for does NOT fire → LightRAG hybrid runs → `confidence=kg
  + error=None + fallback_used=False + markdown_len=0`

The `markdown_len=0` was previously masked by the 60s preempt; now that
retrieval succeeds, the pipeline is silently dropping the synthesizer output
between "kg-confidence chunks retrieved" and "markdown returned to caller".

| H5 | Job-store serializer drops markdown field during JSON encoding | Inspect job-store `update_job(...)` and the `GET /api/synthesize/{job_id}` serializer for filtered fields | Field present in serializer output |

would have blocked landing the timeout fix until the harder diagnosis
completed. Per Q5 (locked): P1 closed = milestone closeable; this phase
