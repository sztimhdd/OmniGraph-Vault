# v1.0.y-3-synth-img â€” `/api/synthesize` image-ref markdown preservation

## Goal

markdown returned by `POST /api/synthesize` (kb-3 Q&A endpoint), so that
when LightRAG-retrieved context contains image markdown, the final
`SynthesizeResult.markdown` stored on the job AND rendered by
`kb/static/qa.js:renderAnswerMarkdown` contains the image refs intact in
the canonical `/static/img/<hash>/<file>` URL form.

- `kb/api_routers/synthesize.py` (POST/GET endpoints â€” likely no change)

## Background

### Symptom (user-reported, scope narrowed 2026-05-25)

When the kb-3 Q&A page (`kb/static/qa.html` + `qa.js`) submits a query
via `POST /api/synthesize` and polls `GET /api/synthesize/{job_id}`,
the rendered answer is missing inline images even though the
LightRAG-retrieved context for the same query contains image markdown
references.

The original v1.0.y-3 plan misdiagnosed this as a Reasoner-stage strip
in the ARAG (b) pipeline. Probe `.scratch/v1.0.y-3-arag-b-probe-20260525-152744.md`
mechanically falsified that hypothesis. Consumer-side reconnaissance
this session showed `kb/static/qa.js:269` calls `/api/synthesize`
(NOT `/api/research`), so the user-visible defect must surface in the
synthesize wrapper, not ARAG.

### Suspect surfaces (probe targets)

Listed in priority order based on initial reconnaissance at HEAD
`6b541232840365a95f6e615a762b35394095cfab`:

1. **Prompt-side (qa template)** â€”
   `kb/services/synthesize.py:_QA_PROMPT_TEMPLATE_ZH/_EN` (lines 136-162)
   instructs the LLM with bare `![alt](URL)` and does NOT pin
   `/static/img/...` or forbid `localhost:8765`. Compare to
   `_LONG_FORM_PROMPT_TEMPLATE_*` (lines 87-122) which DOES pin
   `/static/img/{{hash}}/{{n}}.jpg` and explicitly forbids
   `http://localhost:8765/`. Strong candidate â€” qa-mode answers may
   inherit whatever URL form C1 hands back, and may lose images
   entirely if the LLM treats the loose instruction as optional.

2. **Regex-side (legacy URL rewriter)** â€”
   `kb/services/synthesize.py:_LEGACY_IMAGE_URL_PATTERN` (line 65)
   matches `https?://[^/\s)]+:8765/` only. If C1 emits a different
   absolute URL form (e.g., `http://localhost:9223/...`,
   `http://127.0.0.1:8765/images/...`, or any Hermes-internal IP),
   the rewriter silently passes it through. The browser then 404s,
   which can manifest as "image stripped" if the renderer treats
   broken-image errors specially.

3. **Frontend-side (`qa.js` markdown render)** â€”
   `kb/static/qa.js:renderAnswerMarkdown` (lines 75-89) calls
   `marked.parse(text)` then assigns to `article.innerHTML`. If the
   browser's `Content-Security-Policy` or any post-render sanitizer
   strips `<img>` tags, the user sees no image even when the
   markdown source has the ref. (Lower likelihood since `marked.parse`
   itself doesn't strip; only DOMPurify-style sanitization would.)

4. **Data-side (chunks lack image refs)** â€” possible the LightRAG
   retrieval simply doesn't surface any chunk that contains image
   markdown for the query in question. Probe needs a concrete repro
   query to falsify this; otherwise we cannot reproduce.

### Why a probe report is required before execute

Same discipline as the superseded plan â€” the fix shape depends on
which surface owns the defect. The probe must:

1. Pin HEAD SHA at probe time
2. Identify a concrete reproduction query (specific question that
   retrieves at least one chunk containing `![...](...)` markdown
   in either canonical `/static/img/...` form or any other form)
3. Capture the C1 response body verbatim for that query
   (substring check: does it contain `![`? In what URL form?)
4. Capture the post-`_rewrite_image_urls` markdown stored on the job
5. Capture the rendered DOM (or the `data-md` data-attribute if
   easier than driving Playwright) â€” does the image ref survive
   `marked.parse`?
6. Classify: `prompt-side` / `regex-side` / `frontend-side` /
   `data-side` / `multiple`

### Why S (not M)

The codebase is well-known after the prior session's reconnaissance.
Likely fix shapes:

| Probe outcome | Fix | Size |
| --- | --- | --- |
| `prompt-side`: qa template missing `/static/img/...` pin | Add the pin + the `localhost:8765` ban (mirror long_form template lines 97-100 / 117-119) + regression test | XS-S |
| `regex-side`: `_LEGACY_IMAGE_URL_PATTERN` misses a host form | Broaden the pattern to catch the missed form; add tests | S |
| `frontend-side`: marked.parse / sanitizer strips `<img>` | Investigate sanitizer config; whitelist `<img>` if a sanitizer is in play; otherwise loop back to probe | S-M |
| `data-side`: chunks have no image refs at all | Halt â€” defect is upstream in retrieval or ingestion; convert to follow-up phase | M+ |
| `multiple`: prompt + regex | Two surgical edits + two tests | S |

Conservative t-shirt = **S**. If probe returns `data-side` or
`frontend-side` requiring sanitizer overhaul, halt and escalate.
