# v1.0.y-5 — aim-2/aim-3 cutover residual cache cleanup

## Goal

Clean up the residual cache + backup directories left behind by the
aim-2 (LightRAG storage migration) and aim-3 (systemd timers cutover)
milestones, both of which closed PASS on 2026-05-24:

1. **Aliyun-side stale backup** —
   `/root/OmniGraph-Vault/lightrak_storage.aliyun-pre-aim2-bak`
   (note the `lightrak_` typo, mirroring the canonical `omonigraph`
   typo discipline) was the pre-aim2 snapshot of Aliyun's local
   storage taken right before the byte-identical hydrate from Hermes.
   Since aim-2 verified Aliyun is now byte-identical to Hermes
   (27654 entities / 39604 rels / 1875 chunks / 76249 kv_keys, Δ=0),
   this backup is dead weight on the Aliyun disk and a confusion
   hazard if anyone greps `lightrag_storage` and gets two hits.
2. **Hermes-side cold-backup** —
   `~/.hermes/omonigraph-vault/lightrag_storage` was frozen
   read-only on 2026-05-24 with a 30-day cold-storage hold expiring
   **2026-06-22**. After the hold expires AND aim-3 cron has been
   firing cleanly on Aliyun for ≥2 weeks (post-cutover stability
   window), the Hermes cold-backup can be archived to a tarball and
   the live directory removed from `~/.hermes/omonigraph-vault/`.

This phase is **parallel-safe with no upstream blocker**. Can run
any time alongside v1.0.y-1 / v1.0.y-2 / v1.0.y-3 / v1.0.y-4.

Per Q5 default assumption: this is a P2 phase; if the milestone
window closes with v1.0.y-5 still in flight, it spills to v1.0.z
without blocking milestone close. The Hermes-side cleanup is
**date-gated to 2026-06-22** regardless — if the milestone window
closes before that date, this phase intrinsically defers the Hermes
half to v1.0.z (or whatever phase covers that window).

## Background

### Two distinct cleanup targets, two different gates

| Target | Location | Gate | Scope |
| --- | --- | --- | --- |
| Aliyun stale pre-aim2 backup | `/root/OmniGraph-Vault/lightrak_storage.aliyun-pre-aim2-bak` (typo deliberate per memory `project_aim2_closed_260524.md`) | None — aim-2 closed PASS 2026-05-24, Δ=0 verified | Read-only diagnose → operator-confirm size + last-touched → `rm -rf` |
| Hermes cold-backup live dir | `~/.hermes/omonigraph-vault/lightrag_storage` | **2026-06-22 hold expiry** + aim-3 stability window (≥2 wk clean cron) | Tarball archive (e.g., to `~/.hermes/archive/`) → `rm -rf` of live dir → leave tarball pruned per local retention policy |

The Aliyun side is unblocked today. The Hermes side is calendar-gated
and aim-3-stability-gated; this phase MAY land its Hermes half in a
later v1.0.z run if the calendar gate has not opened by milestone
close.

### Why the Aliyun pre-aim2 backup is safe to delete

aim-2 closure evidence (per memory `project_aim2_closed_260524.md`):

- Aliyun and Hermes byte-identical (entity / relation / chunk / kv
  key counts all Δ=0 against Hermes ground truth)
- Hermes itself is frozen read-only as the upstream truth source for
  the 30-day window
- Any rollback would re-hydrate from Hermes, NOT from the Aliyun
  pre-aim2 backup (which is, by definition, the OLD pre-cutover
  state — rolling back to it would lose 2026-05-24+ ingestion
  activity captured on Aliyun via aim-3 daily cron)

So the pre-aim2 backup is strictly worse than Hermes for any
plausible recovery scenario. It exists only as a "what we had right
before we changed it" artifact, and that role is over.

### Why the Hermes cold-backup needs a 30-day hold

aim-2 cutover policy required Hermes to remain the read-only source
of truth for 30 days after Aliyun went live, in case Aliyun
discovered a hydrate-time corruption or a missing file class that
only manifested under sustained ingestion load. The 2026-06-22
expiry was chosen as the date past which:

- aim-3 daily cron has been firing on Aliyun for ≥4 weeks
- Any ingestion-pattern divergence would have surfaced via
  Reconcile + Postmortem #N
- Hermes the dev box can be reclaimed for other work without
  endangering the production graph

Until 2026-06-22, the Hermes directory MUST NOT be deleted, archived,
or even renamed. This phase's Task 2 (Hermes cleanup) executes only
if the calendar reads 2026-06-22 or later AND aim-3 has been clean
for ≥2 weeks.

| Aliyun grep audit finds live reference to stale backup | halt; do NOT delete; convert to follow-up phase v1.0.y-5-fix that updates the references first |

| Forward-only correction post-commit | new commit `chore(v1.0.y-5): correction — <reason>`. Do NOT amend. |
