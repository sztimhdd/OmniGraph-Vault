# v1.0.y-0 — Add `KB_SYNTHESIZE_TIMEOUT=240` to `databricks-deploy/app.yaml`

## Goal

Land the single-line config drift fix that closes the P0 ghost-success
pattern observed on Databricks `long_form` synthesize: outer
`asyncio.wait_for(timeout=60)` preempts before LightRAG hybrid query can
finish, returning `markdown_len=0 + confidence=no_results +
error="C1 timeout"` at wallclock ~64s.

The fix is a 1-line addition to the env block of
`databricks-deploy/app.yaml` setting `KB_SYNTHESIZE_TIMEOUT=240`, then
a Makefile 6-pass redeploy + post-deploy smoke proving
`markdown_len > 0` on the same `long_form` query that previously hit
the 60s preempt.

This phase is structurally identical to the Aliyun systemd
`override.conf` `Environment=KB_SYNTHESIZE_TIMEOUT=240` that has been
live since `260517-lok-VERIFICATION.md` — Aliyun has the bump,
Databricks never did. The drift is a copy-the-bump exercise, not a
design decision.

Per Q1 (locked): the b-branch probe report does NOT gate this phase
(unrelated path — that is v1.0.y-3 ARAG, separate concern).

Per Q3 (locked): plan is authored in **forward-planning** style with
`status: IN-PROGRESS` and an "Already-done snapshot" block at the top
of Acceptance criteria, citing the verified root cause + Option A fix
from `.planning/quick/260525-c1-no-content-at-64s/REPORT.md`.

**Effective `KB_SYNTHESIZE_TIMEOUT` on the deployed Databricks app: 60 seconds.**

### Smoke evidence already in hand

From `.scratch/260525-kdb-smoke-longform.log`:

| Field | Value |
|---|---|
| `wallclock_s` | 64.19 |
| `confidence` | `no_results` |
| `fallback_used` | True |
| `error_field` | `'C1 timeout'` |
| `markdown_len` | 0 |

Arithmetic match: 60s outer wait_for preempt + ~3-4s `_fts5_fallback`
execution + polling tail ≈ 64.19s. The "C1 timeout" label is
semantically correct (outer wait_for IS firing); the surprise was the
threshold (60s, not the assumed 130s/240s).

### Why Option A (this phase) over Option B (v1.0.y-1)

| Option | Tradeoff | Phase |
| --- | --- | --- |
| **A — env override in app.yaml** | Smallest viable patch; mirrors Aliyun; single-line; fastest unblock; does NOT close the kg_synthesize.py:64-70 comment promise (still no mode dispatch) | **v1.0.y-0 (this phase)** |
| B — wire `mode`-based timeout in kb_synthesize | Structural fix; closes the comment promise; future-proof for new modes; requires test updates; more invasive | v1.0.y-1 (sequential, blocked on this) |
| C — calibrate then set | Lower upper bound on user wait when LightRAG hangs; requires p99 measurement runs; deferred tuning | post-v1.0.y |

A is the unblock; B is the structural follow-up. Sequencing per Q5: P0
ships first to clear the immediate user-visible failure, then v1.0.y-1
re-enters the same module without thrash.

### Cross-environment timeout matrix (REFERENCE)

| Environment | source | Effective | Long-form behavior |
|---|---|---|---|
| **Databricks deploy (PRE-FIX)** | code default | **60s** | wait_for fires t=60s → "C1 timeout" → FTS5 fallback (this phase fixes) |
| **Databricks deploy (POST-FIX)** | app.yaml env | 240s | mirrors Aliyun; sufficient for Databricks `long_form` hybrid wallclock |
| **Aliyun systemd kb-api** | `override.conf` | 240s | unchanged; reference for the 240s bump rationale |
| **Local Windows dev** | `~/.hermes/.env` (unset) | 60s | unchanged; out of scope for this phase |
| **Pytest** | `monkeypatch.setenv("KB_SYNTHESIZE_TIMEOUT", "1")` | 1s | unchanged; test fixtures unaffected |

### Why this is XS

1. Single-line yaml addition to an existing `env:` list (no schema migration, no new env var consumer code).
2. Redeploy via existing `Makefile` 6-pass pipeline (sync → SSG-bake → deploy → wait → log-tail-verify → smoke). Pipeline already proven on prior kdb-2/tk5b deploys.
3. Smoke is a 1-curl `POST /api/synthesize` with `mode=long_form` against the deployed Databricks endpoint, polling `GET /api/synthesize/{job_id}` for `done` + `markdown_len > 0`. Reuses `.scratch/260525-kdb-smoke-longform.log` shape from the diagnostic run.
