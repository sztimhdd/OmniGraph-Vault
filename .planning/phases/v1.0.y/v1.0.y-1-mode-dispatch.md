# v1.0.y-1 — Per-mode `KB_SYNTHESIZE_TIMEOUT` dispatch in `kb_synthesize`

## Goal

Implement structurally-correct per-mode timeout dispatch in
`kb/services/synthesize.py` so that `mode == "long_form"` uses a
240s outer `asyncio.wait_for` budget while `mode == "qa"` retains the
60s budget. This closes the kg_synthesize.py:64-70 comment promise
("long_form requests bump this to 240"), which has been load-bearing
in design intent since 260524-tk5 but never actually wired in
production code.

After this phase lands, `KB_SYNTHESIZE_TIMEOUT` (the legacy single env
var) becomes a deprecated **fallback** for both modes; the canonical
env vars become `KB_SYNTHESIZE_TIMEOUT_QA` (default 60) and
`KB_SYNTHESIZE_TIMEOUT_LONG_FORM` (default 240). Existing callers
that set only the legacy var continue to work (back-compat), but the
default split closes the C1-timeout-drift class of failures
permanently — not just on Databricks (v1.0.y-0) but on every consumer
of `kb_synthesize` (Hermes / Aliyun / Databricks / local).

This is REPORT.md Option B (structural fix). v1.0.y-0 (Option A) lands
first; this phase re-enters the same module with no thrash.

## Background

### Why this phase exists despite v1.0.y-0 already shipping the 240s

v1.0.y-0 patches the symptom for one consumer (Databricks app.yaml
env). The structural defect remains:

1. The `kb_synthesize` function signature accepts `mode: str` (qa |
   long_form) and dispatches the prompt template via
   `_wrap_question_for_mode()`, but the **timeout value passed to
   `asyncio.wait_for` is the module-level `KB_SYNTHESIZE_TIMEOUT`
   constant — independent of mode** (REPORT.md §"Step 2 / point 3",
   referencing `kb/services/synthesize.py:469-541`).
2. The kg_synthesize.py:64-70 comment **lies**: it says "long_form
   bumps to 240" but no code path actually wires the bump.
3. Aliyun has 240s globally (so qa requests there also wait up to
   240s before C1 timeout — wasteful for genuinely-failing qa).
4. Local dev has 60s globally (so long_form locally is unusable
   against Vertex Gemini even when reachability is fine).

The structural fix is to read TWO env vars (with sensible defaults
matching observed-good values) and dispatch by `mode` in
`kb_synthesize`.

### REPORT.md Option B sketch (verbatim, lines 269-280)

```python
KB_SYNTHESIZE_TIMEOUT_QA: int = int(os.environ.get("KB_SYNTHESIZE_TIMEOUT_QA", "60"))
KB_SYNTHESIZE_TIMEOUT_LONG_FORM: int = int(os.environ.get("KB_SYNTHESIZE_TIMEOUT_LONG_FORM", "240"))

# in kb_synthesize, line ~523:
timeout = KB_SYNTHESIZE_TIMEOUT_LONG_FORM if mode == "long_form" else KB_SYNTHESIZE_TIMEOUT_QA
response = await asyncio.wait_for(
    synthesize_response(query_text, mode="hybrid"),
    timeout=timeout,
)
```

### Back-compat tier

Existing `KB_SYNTHESIZE_TIMEOUT` env (legacy) MUST remain functional
to avoid breaking Hermes / Aliyun / local-dev configs that set it.
Resolution order:

```
mode == long_form: KB_SYNTHESIZE_TIMEOUT_LONG_FORM > KB_SYNTHESIZE_TIMEOUT > 240 (default)
mode == qa:        KB_SYNTHESIZE_TIMEOUT_QA        > KB_SYNTHESIZE_TIMEOUT > 60  (default)
```

This way:

- Aliyun's existing `Environment=KB_SYNTHESIZE_TIMEOUT=240` (systemd
  override) keeps both modes at 240s — same as today, no regression.
- Databricks app.yaml `KB_SYNTHESIZE_TIMEOUT=240` (just shipped in
  v1.0.y-0) keeps both modes at 240s — same as right after v1.0.y-0,
  no regression.
- New deployments can opt into the split by setting
  `KB_SYNTHESIZE_TIMEOUT_QA=60` and
  `KB_SYNTHESIZE_TIMEOUT_LONG_FORM=240` explicitly.
