# v1.0.y-4 — Reconcile `articles.lang` schema drift between Aliyun prod and `.dev-runtime/` mirror

## Goal

Bring the `articles.lang` column (or whatever the canonical name turns
out to be — `lang` / `language` / absent / nullable / typed) into
parity between:

- **Prod (Aliyun)** — `/root/OmniGraph-Vault/data/kol_scan.db` (the
  ground-truth ingestion DB powering the daily systemd-timer cron)
- **Dev mirror** — `.dev-runtime/data/kol_scan.db` (the local Windows
  dev box copy used for `.scratch/local_serve.py` UAT and `local_e2e.sh`
  smoke runs)

So that any code change touching language detection, RSS classifier, or
KB synthesis lang routing exercises the same column shape locally as
it does in prod, and `local_e2e.sh` smoke runs do not silently mask a
schema-drift bug (the v1.0.x ingestion-orchestration class — see Project
CLAUDE.md "Lessons Learned 2026-05-15 #2").

This phase is **parallel-safe with no upstream blocker**. Can run any
time alongside v1.0.y-1 / v1.0.y-2 / v1.0.y-3.

Per Q5 default assumption: this is a P2 phase; if the milestone window
closes with v1.0.y-4 still in flight, it spills to v1.0.z without
blocking milestone close.

## Background

### Symptom

The drift was first surfaced in passing during v1.0.y-2 (zh-CN FTS5
zero-hit) planning: the v1.0.y-2 plan references "articles.lang
schema drift — see v1.0.y-4" because FTS5 hypothesis 1 (CJK tokenizer)
needs a stable language label to verify which articles SHOULD be
indexed under what tokenizer. Without column parity, any local
reproduction fixture either (a) crashes on `KeyError: 'lang'` or
(b) silently substitutes a default value, masking the prod behavior.

The drift is also implicated, indirectly, in the 260522-clt body
translation pass (Project CLAUDE.md memory
`project_260522_clt_pass2_translation.md`): the `body_translated`
gap close-out depended on accurately classifying which articles were
zh-CN vs en-US. If the dev mirror has `lang IS NULL` everywhere
while prod has populated values, any local dry-run of the translation
batch reads the wrong candidate count.

### Why this is non-trivial

The exact shape of the drift is unknown until Task 1 runs the
side-by-side schema dump. Possible drift shapes (any one or
combination):

| # | Drift shape | Likelihood | Reconcile size |
| --- | --- | --- | --- |
| 1 | Column `lang` exists in prod but NOT in dev (dev is from a pre-lang migration) | High | Add column + backfill from dev's existing data |
| 2 | Column exists in both but with different name (`lang` vs `language`) | Low | Rename one to match the other |
| 3 | Column exists in both with same name but different NULL/NOT-NULL constraint | Medium | Loosen or tighten dev to match prod |
| 4 | Column exists in both but populated in prod (n=hundreds), empty/NULL in dev | High | Backfill dev from prod export (or run dev-side classifier offline) |
| 5 | Column exists in prod with a CHECK constraint (e.g., `IN ('zh','en','mixed')`); dev is unconstrained | Medium | Add CHECK to dev |
| 6 | A migration script lives in `scripts/migrations/` that prod ran but dev never did | High | Run the migration on dev |

These are not mutually exclusive. Task 1 produces the canonical drift
report; Task 2 applies the reconcile.

### Why dev → prod, not prod → dev

Aliyun prod is the ground truth of ingest activity. Dev mirror is a
read-only-ish substrate for local UAT + smoke. We bring dev INTO LINE
with prod, never the other way around. If a desired schema change is
identified during Task 1, that becomes a **separate forward-only
schema-migration phase** under v1.0.z or later — NOT inside this
phase. This phase is reconcile-only.

Per HARD CONSTRAINT awareness (Project CLAUDE.md): no mutating ops on
Aliyun in this phase. `aliyun-vitaclaw` SSH is read-only diagnosis.

### Why S (not XS)

| Drift shape | Reconcile shape | Size |
| --- | --- | --- |
| Single missing column, simple `ALTER TABLE` | XS | XS |
| Missing column + backfill from a deterministic source | S | S |
| Column exists but populated differently — needs export-from-prod-import-to-dev | S | S |
| Multiple drifts combined (column + CHECK + migration script not run) | M | escalate to v1.0.z |

Conservative t-shirt = S. If Task 1 reports M-tier (multi-drift +
migration replay needed), halt + escalate to a follow-up phase
v1.0.y-4-fix.

## Acceptance criteria

1. Task 1 produces `.scratch/v1.0.y-4-schema-diff-<ts>.log` with:
   - Aliyun prod schema dump for `articles` table
     (`SELECT sql FROM sqlite_master WHERE name='articles'`)
   - Local dev schema dump for the same
   - Side-by-side diff highlighting every column-level discrepancy
     (name, type, NULL constraint, default, CHECK constraint)
   - Row-count parity check
     (`SELECT COUNT(*), SUM(CASE WHEN lang IS NULL THEN 1 ELSE 0 END) FROM articles`
     on both, where `lang` is the canonical column name once
     identified)
   - List of `scripts/migrations/*` files mentioning `lang` /
     `language` and which (if any) ran on prod but not dev
2. If Task 1 classifies as M-tier (multiple drifts + migration replay):
   **HALT + escalate**; do NOT proceed to Task 2. Convert to follow-up
   phase v1.0.y-4-fix.
3. If Task 1 classifies as S/XS: proceed to Task 2.
4. Task 2 applies reconcile via the smallest possible diff:
   - For "missing column" drift: `ALTER TABLE articles ADD COLUMN lang
     TEXT` (or matching prod's exact type/constraint)
   - For "missing migration" drift: run the existing
     `scripts/migrations/<NNN>_*.py` (or `.sql`) against
     `.dev-runtime/data/kol_scan.db`
   - For "values diverged" drift: export prod column via
     `sqlite3 ... ".dump articles"` filtered to `(article_hash, lang)`
     pairs; apply via parameterized UPDATE on dev
   - All operations atomic (temp table or transaction); pre-op backup
     of `.dev-runtime/data/kol_scan.db` to
     `.dev-runtime/data/kol_scan.db.bak-pre-v1.0.y-4-<ts>`
5. Post-reconcile verification: re-run the Task 1 schema diff; confirm
   ZERO column-level discrepancies and matching populated-row counts
   (within ±1% tolerance for in-flight prod ingest activity).
6. No production code (under `kb/`, `lib/`, `enrichment/`, root scripts)
   is touched. This phase is dev-DB-only reconcile + evidence.
7. `local_e2e.sh layer1 5` and `local_e2e.sh layer2 5` smoke runs
   complete without `KeyError`/`OperationalError` on the `lang` column.
   Captured to `.scratch/v1.0.y-4-local-smoke-<ts>.log`.
8. No regression on Aliyun prod: confirmed by grep — no Aliyun-side
   files / SQL touched in this phase.
9. Single forward-only commit on `main`. Commit message:
   `chore(v1.0.y-4): reconcile articles.lang schema drift dev↔prod — <case-classification>`.
   Body cites Task 1 schema-diff log + Task 2 reconcile log per HARD
   CONSTRAINT #6.
10. `git status` clean post-commit.
