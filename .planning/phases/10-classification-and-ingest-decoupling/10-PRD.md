# Phase 10 PRD — Scrape-First Classification + Text-First Ingest Decoupling

**Milestone:** v3.1 Single-Article Ingest Stability
**Requirements covered:** CLASS-01, CLASS-02, CLASS-03, CLASS-04, ARCH-01, ARCH-02, ARCH-03, ARCH-04 (8 REQs — largest phase in milestone)
**Dependencies:** Phase 8 (image pipeline correctness) + Phase 9 (timeout + state management with `get_rag(flush)` + rollback)
**Downstream blockers:** Phase 11 E2E gate depends on the entire decoupled pipeline working end-to-end

---

## Goal

Replace unreliable digest-based classification with full-text classification AND decouple Vision (image description) from the text-ingest hot path. After this phase:

- A classifier decision is based on article body, not a `digest` field that's often N/A or truncated
- The `ainsert` call for the article text returns within seconds (no Vision API wait)
- Image descriptions are generated by a background worker and linked back to the graph as **append sub-docs** (decided in REQUIREMENTS.md ARCH-03 — not re-embed)
- Graph is queryable immediately after text ingest

---

## Locked Decisions (from REQUIREMENTS.md — these ARE acceptance criteria)

### CLASS-01 — Scrape before classify
`batch_ingest_from_spider.py --from-db` flow (and sibling orchestrators) scrape the full article body BEFORE classification. NO reliance on WeChat `digest` field.

### CLASS-02 — DeepSeek classifies on full body
DeepSeek called with full article text; returns `{depth: 1-3, topics: [...], rationale: str}`.

### CLASS-03 — Reuse existing WeChat anti-abuse params
Reuse `spiders/wechat_spider.py`: `SESSION_LIMIT=54`, `RATE_LIMIT_SLEEP_ACCOUNTS=5.0`, `RATE_LIMIT_COOLDOWN=60.0`, random shuffle, existing `_UA_POOL` / `_ua_cooldown()`. Do NOT invent new rate-limit parameters. **v3.1 gate runs against local fixture — WeChat rate limit is BATCH-scenario-only, does not block Phase 11 gate.** This REQ is for spec-correctness of the batch path Phase 5 will invoke.

### CLASS-04 — Persist classifier output to SQLite
`classifications` table receives `article_id`, `depth`, `topics` (JSON-serialized), `rationale`, `classified_at` BEFORE ingest decision is made.

### ARCH-01 — Text ingest is synchronous-fast
Article body + image file paths `ainsert`-ed FIRST. Produces queryable graph immediately. Vision/image-description does NOT block this path.

### ARCH-02 — Async Vision worker
After text ingest returns, image descriptions are generated by a background (async) worker. Spawn the worker via `asyncio.create_task()` or explicit task queue — design detail up to planner.

### ARCH-03 — Append sub-doc (LOCKED — not re-embed)
Vision-generated descriptions are linked to the graph by **appending an image sub-doc via `ainsert`** (one sub-doc per image OR per article — planner's call, but single sub-doc per article is simpler). Sub-doc's `ids=[f"wechat_{article_hash}_images"]` and content is a markdown list of `[image N] <description>` lines referencing back to the parent. Rationale from REQUIREMENTS.md: re-embed would invalidate stable vectors + cascade updates; append-sub-doc is incremental and fails independently.

### ARCH-04 — Vision failure does not invalidate text ingest
SiliconFlow + OpenRouter + Gemini Vision all down → text ingest still succeeds, graph still queryable. The async Vision worker's failure is logged + swallowed; does NOT rollback text ingest.

1. Classifier test: pass a fixture article with `digest=""` (or `digest=N/A`) → classifier reads full body → returns `{depth: 1, topics: ["news"], rationale: "..."}` for the GPT-5.5 news fixture (should NOT be depth=3 anymore)

## Implementation Notes (planner's discretion)

- **Scrape-first CLI flow:** Currently `batch_ingest_from_spider.py --from-db` reads from already-scraped articles. Extend: when the `articles.body` column is empty for a pending article, scrape on-demand (via `spiders.wechat_spider` which already has rate limiting), write body to DB (or in-memory), then classify.
- **Classifier prompt:** reuse `batch_classify_kol.py` prompt structure but feed full body instead of digest. Same JSON shape expectation.
- **Text ingest `ainsert`:** call with parent `doc_id = f"wechat_{article_hash}"` (already established in Phase 9). `ids=[doc_id]`.
- **Image sub-doc `ainsert`:** call with `doc_id = f"wechat_{article_hash}_images"`. Content:
  ```
  # Images for <title>
  
  - [image 0]: <description>
  - [image 1]: <description>
  ...
  ```
- **Background worker:** spawn via `asyncio.create_task()` returned from `ingest_article`. Orchestrator is free to `await task` later for tests or let it fire-and-forget in production.
- **Vision failure policy (ARCH-04):** inside the async worker, wrap the describe_images call in try/except. Write partial descriptions if some images succeeded; log the aggregate `image_batch_complete` with failure counts per D-08.02.
- **Integration tests:** may be slow due to real `ainsert` + LightRAG internals. Gate individual integration tests with `@pytest.mark.integration` and consider mocking LightRAG's `ainsert` + `adelete_by_doc_id` for logic-level tests.
