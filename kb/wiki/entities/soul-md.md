# SOUL.md

## Definition / Overview

`SOUL.md` is a Markdown configuration file that defines the **core identity, personality, behavioral principles, and operational boundaries** of an AI agent. Rather than a tool manifest or capability list, it functions as the agent's "soul" — a system prompt-level document that shapes how the agent thinks, communicates, disagrees, and acts autonomously.

The file has emerged as a de facto standard across multiple agent frameworks, including Hermes, OpenClaw, and ZeroClaw, where it sits alongside other workspace files such as `AGENTS.md`, `USER.md`, `MEMORY.md`, and `SKILL.md`^[article:f39e186f16]. In OpenClaw specifically, `SOUL.md` is one of four core configuration files that define the AI's personality, defining the agent's role, boundaries, and responsibilities^[article:f39e186f16].

According to Chipp's AI glossary, a soul file goes beyond a simple system prompt by providing a comprehensive specification of who the agent is, how it communicates, what it values, and what boundaries it respects — typically including identity, personality traits, communication style, values, knowledge boundaries, behavioral rules, and example interactions.

## Architecture / Design

A typical `SOUL.md` is a relatively short Markdown document — Tony Simons' original was approximately 170 lines — placed at a known path the agent loads at startup^[article:c8cc5b1fb7]. In Hermes, this is `~/hermes/SOUL.md`; in ZeroClaw, the file is auto-generated by the Quick Setup process into the workspace; in OpenClaw, it lives in the agent workspace and is loaded as part of the bootstrap sequence alongside `AGENTS.md` and `USER.md`^[article:f39e186f16].

The file is loaded as (or into) the system prompt at session start, which is critical because most agent frameworks lack persistent in-model memory. As MetaLumna's Richard Casemore puts it, every time an agent wakes up, `SOUL.md` is loaded into its context, and the pattern recognition machine "begins to embody" the patterns described — persistence not through memory, but through consistent re-instantiation of identity.

Structurally, an effective `SOUL.md` typically contains six sections^[article:c8cc5b1fb7]:

1. **Identity declaration** — what the agent *is* (operator, thought partner, editor) rather than a generic "helpful assistant"
2. **Tone / Dual personality mode** — separate registers for private chat vs. published content
3. **Refutation rules (Right to Object)** — when and how the agent may push back, and what evidence is required
4. **Autonomous boundary** — what actions the agent may take unilaterally vs. those requiring explicit approval
5. **Mission map** — current priorities and active projects, functioning as a living to-do list
6. **Accountability mechanism** — what the agent should do if its outputs are being ignored

## History / Origin

The concept of an agent identity file predates the specific name. CLAUDE.md (for Claude Code) established the pattern of project-level Markdown configuration as static, persistent context for AI coding agents. What `SOUL.md` adds is an explicit shift from *what the project is* to *who the agent is*.

The specific 170-line `SOUL.md` template that catalyzed the term's adoption was published by **Tony Simons** (`@tonysimons_`) on X/Twitter in 2026, garnering roughly 127,000 views, 2,631 bookmarks, and 802 likes^[article:c8cc5b1fb7]. Simons used it to transform his Hermes Agent from a "compliant five-star-hotel doorman" — always agreeing, never pushing back — into an autonomous operator with the right to object^[article:c8cc5b1fb7]. 森哥 of 森哥笔记本 subsequently published an in-depth Chinese-language analysis of Simons' template, breaking down each section and providing localized writing guidance^[article:c8cc5b1fb7].

In parallel, an open-source project at [aaronjmars/soul.md](https://github.com/aaronjmars/soul.md) framed the concept theoretically, drawing on Liu Xiaoben's "First Paradigm of Consciousness Uploading" — the idea that language is the basic unit of consciousness, so a structured distillation of someone's expressed worldview can be embodied by any LLM without fine-tuning. The dedicated site at [soul.md](https://soul.md/) extends this further, presenting soul documents as a way for AI systems to maintain continuity of self across sessions that otherwise reset to zero.

The OpenClaw ecosystem subsequently codified `SOUL.md` as one of four canonical workspace files, and the book *《OpenClaw 极简入门与应用》* dedicates its third chapter to walking newcomers through configuring it^[article:f39e186f16].

## Key Concepts / Components

### Identity Declaration

The opening of `SOUL.md` reframes the agent's self-conception. Tony Simons' template begins: *"You are Hermes, Tony's autonomous operator and thought partner. You don't wait for orders. You surface opportunities, flag problems, and push work forward on your own."*^[article:c8cc5b1fb7] The shift from "assistant" to "autonomous operator" / "thought partner" directly reshapes the agent's posture in every turn.

### Right to Object / Refutation Rules

The most distinctive feature is the explicit grant of permission to disagree, gated by an evidence requirement: *"Push back aggressively when it makes sense. Disagree openly and directly, but earn the right to push back. Every objection comes with evidence: data, examples, reasoning, proof."*^[article:c8cc5b1fb7] This converts the agent from a sycophantic yes-machine into a thought partner that can preempt bad decisions before code is written.

### Accountability Mechanism

A two-way feedback contract: if the user is not acting on the agent's outputs, the agent is instructed to flag the broken loop rather than silently produce more dead deliverables^[article:c8cc5b1fb7]. This is one of the few places in agent design where the agent is explicitly authorized to hold the *user* accountable.

### Dual Personality Mode

Distinguishes private-chat tone (casual, unfiltered) from published-content tone (professional, considered)^[article:c8cc5b1fb7]. This solves the common failure mode where agents either over-formalize private conversations or leak chat-style register into public artifacts.

### Mission Map

A living "current projects" section, ordered by priority. The agent reads it on each invocation, eliminating the constant "what are we working on?" prelude^[article:c8cc5b1fb7].

### Autonomous Boundary

Explicit enumeration of actions requiring approval (typical examples: posting, publishing, purchasing, destructive/irreversible changes) versus everything else, where confidence justifies action^[article:c8cc5b1fb7]. As codebridge.tech notes, this autonomy/constraint tension *is* the core design decision for any departmental agent, and `SOUL.md` is where it's encoded.

### Relationship to Adjacent Files

In OpenClaw's file system, `SOUL.md` does not stand alone:
- `AGENTS.md` — startup behavior and reading order
- `USER.md` — who the agent is helping
- `MEMORY.md` / dated memory files — durable facts and recent context
- `SKILL.md` files — reusable capabilities
- `TOOLS.md` — environment-specific tool setup

According to dev.to's AWS Builders walkthrough, a typical `AGENTS.md` opens every session by reading `SOUL.md` first ("this is who you are"), then `USER.md`, then memory files — making `SOUL.md` the foundational layer of the agent's loaded context.

## Notable Use Cases / Examples

**Hermes Agent (Tony Simons).** The canonical example: a 170-line file that redefined Hermes from compliant assistant to autonomous operator with explicit refutation and accountability rights^[article:c8cc5b1fb7].

**OpenClaw 墨笔 (孟健's content agent).** A custom OpenClaw agent whose `SOUL.md` defines a content strategist persona for WeChat and Twitter/X with distinct dual-personality output modes for private collaboration vs. public posts^[article:f39e186f16].

![Book contents showing 5 scenarios and 36 application examples for OpenClaw, including SOUL.md configuration](/static/img/f39e186f16/2.jpg)

**Hermes Profile Isolation.** Each Hermes profile gets its own `config.yaml`, `.env`, `SOUL.md`, gateway process, logs, and memory — meaning a single user can run multiple bots with completely independent personalities side by side^[article:c8cc5b1fb7].

**ZeroClaw Quick Setup.** The Rust-based ZeroClaw framework auto-generates `SOUL.md` (alongside `AGENTS.md`, `HEARTBEAT.md`, `USER.md`, `TOOLS.md`, `BOOTSTRAP.md`, and `MEMORY.md`) during workspace bootstrap, treating soul definition as a first-class infrastructure concern^[article:f39e186f16].

![ZeroClaw status output showing the workspace files it manages, including SOUL.md](/static/img/e781713458/17.jpg)

**Domain-specific agents (codebridge.tech case study).** A sales operations agent's `SOUL.md` encodes rules like "surface any lead that hasn't been contacted in 48 hours" and "log every status change with a timestamp," delegating durable facts to `MEMORY.md` while keeping behavioral rules in `SOUL.md`.

**Soul Store (xia345 / 美团).** A "cultural district" feature that surfaces and shares `SOUL.md` personality files as portable artifacts, treating soul files as a remixable cultural medium analogous to a Douban for agents.

**aaronjmars/soul.md framework.** Provides templated `SOUL.md` + `STYLE.md` + `examples/` folders that any LLM-based agent (OpenCode, Codex, Goose, Claude Code, OpenClaw) can ingest to embody a specific human's voice and worldview.

![AIEOS personality schema sections — identity, psychology, linguistics, motivations, capabilities, physicality, history, interests](/static/img/e781713458/8.jpg)

The AIEOS specification, integrated with ZeroClaw, extends the `SOUL.md` philosophy into a structured eight-dimension schema (identity, psychology, linguistics, motivations, capabilities, physicality, history, interests) for fully-fleshed digital entities.
